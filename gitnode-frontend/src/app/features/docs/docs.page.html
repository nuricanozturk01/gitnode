HTTP / HTTPS Git

Smart Git is served on the same origin as the REST API under /git/<owner>/<repo>. Git sends HTTP Basic auth: use your GitNode username and account password when prompted.

Path tip: Do not add a .git suffix in the URL path — storage resolution appends it on the server. With TLS in front of the app, use https:// instead of http://.

Actions — Overview

GitNode Actions is a CI/CD engine built into GitNode. Define workflows in {{ actionsWorkflowPath }} files and they run automatically on push, pull request opened, or manual trigger. No external services — everything runs on runners you control.

Key differences from GitHub Actions

Feature GitHub Actions GitNode Actions
Workflow location .github/workflows/ .gitnode/workflows/
Action marketplace Any action from marketplace Built-in actions only (no marketplace)
Runners GitHub-hosted or self-hosted Self-hosted only (register your own)
Composite / reusable workflows Supported Not supported
Built-in context (github.*) ${{'${{ github.sha }}'}} in expressions and env ${{'${{ github.sha }}'}} in expressions; shell env uses GITNODE_* vars
Secrets syntax ${{'${{ secrets.NAME }}'}} ${{'${{ secrets.NAME }}'}} (same)
Matrix strategy Supported Supported
Concurrency Supported Supported
Artifacts Stored by GitHub Stored in GitNode backend

Auto-injected environment variables

Every job receives these shell environment variables when the run provides the underlying data. Use them in run: scripts. Expression context (${{'${{ github.sha }}'}}, ${{'${{ inputs.name }}'}}) is resolved server-side before dispatch.

Variable Description
GITNODE_OWNER Repository owner username
GITNODE_REPO Repository name
GITNODE_SHA Commit SHA when the run recorded one (not set on every push trigger today)
GITNODE_REF Branch or tag ref (e.g. refs/heads/main)

Step 2 — Create a workflow file

Create {{ actionsWorkflowPath }} in your repository: 

{{ actionsMinimalYaml }}

Step 3 — Push and watch

Commit and push to main. Navigate to Actions → All runs in your repository to see the run appear. Click a run to follow step-by-step logs in real time via SSE streaming.

What the runner needs installed: The actions/checkout@v1 step clones your repo via authenticated HTTP — no manual Git credential setup required. Trigger a workflow on demand from the UI without pushing code. Add workflow_dispatch: to your trigger list and optionally define typed input fields.

How to trigger

  2. Find the workflow and click Run workflow.
  3. Choose the target branch and fill in any declared inputs.
  4. Click Run workflow — the run appears in All runs immediately.

Using secrets in steps

Reference secrets with ${{'${{ secrets.NAME }}'}} in step definitions. GitNode resolves them before dispatching the job — values are decrypted server-side (AES-256-GCM) and masked in log output. Secrets require ACTIONS_ENCRYPTION_KEY to be configured on the server. 

{{ actionsSecretsYaml }}

Manage secrets at Repository → Settings → Actions (repo owner only). Each secret is stored encrypted; the value is never shown after creation.

Self-hosted runners

Every job runs on a runner you register. Runners connect via WebSocket, receive jobs, and stream logs back in real time.

How runners work

  1. Generate a registration token in your account Settings → Actions (runners are account-scoped, not per repository).
  2. Start the runner binary with --token and --labels. It registers itself and gets a long-lived JWT.
  3. The runner opens a persistent WebSocket to GitNode and sends heartbeats every 30 s.
  4. When a workflow triggers a job, GitNode picks the first idle runner whose labels match runs-on: and dispatches the job.
  5. The runner executes steps in order, streams logs live, and reports completion.

Runner statuses

Status Meaning
online Idle and ready to accept jobs
busy Currently executing a job
offline Not connected or no heartbeat in 60 s

Matching runners with runs-on:

A runner is selected when all labels in runs-on: are present on the runner. Register multiple runners with different label sets to route jobs to specific machines (e.g. linux, gpu, arm64).

Start command

{{ actionsRunnerCmd }}

--executor can be shell (default) or docker. The registration token expires after 1 hour and is consumed on first registration; the same token can re-register an existing runner. The runner JWT it generates is valid for 1 year.

Collaborators

Invite other GitNode users with fine-grained per-permission roles — each permission toggled independently, so you grant exactly the access each person needs.

Available permissions

Permission What it allows
READ Clone and view the repository (required for private repos)
PUSH Push commits and branches
PULL_REQUEST_CREATE Open pull requests
PULL_REQUEST_REVIEW Review and comment on pull requests
PULL_REQUEST_MERGE Merge and close pull requests
ISSUE_MANAGE Edit, close, and delete any issue or comment (including others' content). Creating issues and commenting only requires repo access.
ACTIONS_WRITE Trigger manual workflow runs, cancel runs, and delete workflow runs
SETTINGS_READ View the repository Settings page
SETTINGS_WRITE Modify repository settings and webhooks. Actions secrets remain repo-owner-only.
ADMIN All of the above

Inviting a collaborator

  1. Open your repository → Settings → Collaborators.
  2. Enter the invitee's existing GitNode username, toggle permissions, and click Invite.
  3. For pending collaborators, click Invite Link to generate a shareable URL (not created automatically on invite).
  4. The recipient opens the link and accepts — no approval from you required.
Notes:
  • Invite links expire after 7 days — generate a new one if it has expired.
  • Private repositories remain private to everyone who is not a collaborator or owner.
  • Revoke access at any time from Settings → Collaborators.

GitHub Repository Migration

Import a GitHub repository into your GitNode account — full commit history, and optionally pull requests and tags/releases — in a single form.

Migration options

Item Details
Repository Full commit history mirror-cloned from the GitHub HTTPS URL — always included
Pull Requests (optional) Imported with titles, descriptions, and branch names. Open GitHub PRs stay open; closed GitHub PRs are closed locally.
Tags & Releases (optional) Tags and release metadata imported from GitHub

How to migrate

  1. Open the Import repository page in GitNode (from the dashboard or new repository flow).
  2. Enter the full GitHub HTTPS URL, e.g. https://github.com/owner/repo.
  3. Paste a GitHub Personal Access Token with repo read access.
  4. Select which optional items to migrate (pull requests, tags & releases), then click Start migration.
  5. GitNode runs the import in the background and returns you to the dashboard. Optional PR and tag/release imports may continue asynchronously after the job is marked complete.
Important:
  • Migration is a one-time snapshot — there is no ongoing sync with GitHub.
  • Private GitHub repositories require a PAT with private repository read scope.
  • The mirror clone sets the default branch to main regardless of GitHub's default branch name.

Project Management (Kanban)

Organize work with Kanban boards linked to your repositories. Tasks connect directly to branches and pull requests for a seamless workflow.

Structure

Level Description
Project Top-level container with optional repository links (a project can link multiple repos); has its own settings and automation rules
Board A view inside a project with configurable columns (e.g. To Do, In Progress, Done)
Task A unit of work with type, status, assignee, and ordering; can hold subtasks
Subtask A finer-grained step inside a task with its own status and assignee

Branch and pull request integration

  • Create a branch from a task — GitNode generates a conventional branch name from the project's code prefix, task number, and title (e.g. PROJ-1-add-login-page) and creates it in the linked repository.
  • Subtask branches use the format PROJ-1.SUB-1-fix-validation (prefix comes from your project settings).
  • Link a PR to a task or subtask — the board shows a PR indicator; open/merged/closed status is visible on the task detail page.
  • PR merge → task auto-complete — enabled by default per project; toggle in Project → Settings. When a linked PR is merged, GitNode marks the task status as completed (the card stays in its current column).
Where to find projects: Repository → Projects tab or the top-level navigation. Columns are configured per project in Project → Settings.

Webhooks

Receive signed HTTP POST requests on repository events — pushes, branches, pull requests, issues, project tasks, snippets, and workflow runs. Includes automatic retries, a dead-letter queue, and per-host circuit breakers.

Delivery

Each delivery is a POST with Content-Type: application/json. When a webhook secret is set, GitNode adds an HMAC-SHA256 signature header. The event type and delivery metadata are in the JSON payload body.

Header Value
Content-Type application/json
X-Hub-Signature-256 sha256=<HMAC-SHA256 hex> — only sent when a webhook secret is configured; verify this before processing the payload

Verifying the signature (Python)

{{ webhookVerifyPython }}

Reliability

Feature Behaviour
Automatic retries 3 attempts with exponential back-off (1 s, 2 s, 4 s) on non-2xx or connection errors
Dead-letter queue Permanently failed deliveries land in the DLQ and are retried on a schedule; platform admins can inspect and replay from the admin panel
Circuit breaker Per-host Resilience4j circuit breaker — repeated failures open the circuit; deliveries go straight to DLQ until the endpoint recovers
Setup: Configure webhooks per repository in Settings → Webhooks.

Repository Forks

Fork any public repository and work on it independently under your account. Pull requests are same-repository only — cross-repo PRs from a fork are not supported.

Fork workflow

  1. Navigate to any public repository and click Fork in the repository header.
  2. GitNode creates a copy under your account with the full commit history at the time of forking.
  3. Clone your fork, create a branch, make your changes, and push to your fork.
  4. To contribute upstream, open a pull request in the upstream repository using branches that exist there, or ask the upstream owner to pull from your fork manually.
Note: Forks are not synced with upstream automatically — rebase or merge manually to pull in upstream changes.

Enterprise SSO — SAML & LDAP

Map an organization's email domain to a SAML 2.0 IdP or corporate LDAP directory. Users sign in with their work email and are provisioned automatically on first login.

Global feature flags

Environment variable Default Purpose
GITNODE_SSO_SAML_ENABLED false Enable the SAML 2.0 SP globally
GITNODE_SSO_LDAP_ENABLED false Enable LDAP authentication globally
GITNODE_SSO_SAML_SP_SIGNING_KEY_PATH — Path to SP signing private key (required for SAML)
GITNODE_SSO_SAML_SP_SIGNING_CERT_PATH — Path to SP signing certificate (required for SAML)

SAML 2.0 setup

  1. Generate the SP key pair: 
    {{ samlKeygenCmd }}
  2. Set GITNODE_SSO_SAML_ENABLED=true and point the key/cert env vars to the generated files.
  3. In Admin Panel → Organizations, open the org, paste the IdP metadata URL or XML, test the connection, then enable.
  4. Set the org's email domain — users who enter it on the login page will be routed to SAML.

LDAP setup

  1. Set GITNODE_SSO_LDAP_ENABLED=true.
  2. In Admin Panel → Organizations, configure the org's LDAP: server URL, manager bind DN and password, user search base, filter, and attribute mappings for email and display name.
  3. Click Test connection before enabling.
  4. Set the org's email domain so the login page routes matching addresses to LDAP.
Important rules:
  • SAML and LDAP are mutually exclusive per organization — enable one or the other, not both.
  • New accounts are provisioned automatically on first successful SSO login.
  • Use make ldap-up to start a test OpenLDAP container locally.
