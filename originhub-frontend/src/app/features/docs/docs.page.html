HTTP / HTTPS Git

Smart Git is served on the same origin as the REST API under /git/<owner>/<repo>. Git sends HTTP Basic auth: use your OriginHub username and account password when prompted.

Path tip: Do not add a .git suffix in the URL path — storage resolution appends it on the server. With TLS in front of the app, use https:// instead of http://. 
{{ httpsCloneLine() }}

Smart HTTP can be turned off with originhub.http.enabled=false on the server — then use SSH only.

Actions — Overview

OriginHub Actions is a CI/CD engine built into OriginHub. Define workflows in {{ actionsWorkflowPath }} files and they run automatically when you push code, open a pull request, or trigger them manually from the UI. No external services needed — everything runs on runners you control.

Key differences from GitHub Actions

Feature GitHub Actions OriginHub Actions
Workflow location .github/workflows/ .originhub/workflows/
Action marketplace Any action from marketplace Built-in actions only (no marketplace)
Runners GitHub-hosted or self-hosted Self-hosted only (register your own)
Composite / reusable workflows Supported Not supported
Context expressions ${{'${{ github.sha }}'}} ORIGINHUB_SHA env var
Secrets syntax ${{'${{ secrets.NAME }}'}} ${{'${{ secrets.NAME }}'}} (same)
Matrix strategy Supported Supported
Concurrency Supported Supported
Artifacts Stored by GitHub Stored in OriginHub backend

Auto-injected environment variables

Every job automatically receives these variables. Use them in run: scripts or step env: blocks.

Variable Description
ORIGINHUB_OWNER Repository owner username
ORIGINHUB_REPO Repository name
ORIGINHUB_SHA Full commit SHA that triggered the run
ORIGINHUB_REF Branch or tag ref (e.g. refs/heads/main)
ORIGINHUB_REPO_ID Repository UUID (used by cache and artifact actions)

Manual dispatch

Trigger a workflow on demand from the UI without pushing code. Add workflow_dispatch: to your trigger list and optionally define input fields that appear in the "Run workflow" dialog.

How to trigger from the UI

  1. Open your repository and navigate to Actions → Workflows.
  2. Find the workflow and click the Run workflow button.
  3. A dialog appears where you choose the target branch and fill in any declared inputs.
  4. Click Run workflow — the run appears in All runs immediately.

Supported input types

Type UI control YAML value
string (default) Text field Any string
choice Dropdown — options from options: list One of the declared options
boolean Toggle switch "true" or "false"

Accessing inputs in steps

Use ${{'${{ inputs.name }}'}} anywhere in a step's run:, with:, or env: block. OriginHub resolves these expressions before sending the job to the runner — the raw ${{'${{ }}'}} syntax never reaches the shell. Inputs not provided by the user resolve to their default: or an empty string.

Built-in actions

OriginHub ships a set of built-in actions. There is no external marketplace — every uses: reference must match one of these.

actions/checkout@v1 required first step

Clones the repository into the job workspace using the authenticated runner token. Reads ORIGINHUB_OWNER and ORIGINHUB_REPO from the injected env vars automatically — no configuration needed.

Input Default Description
ref ORIGINHUB_SHA Ref to checkout (branch, tag, SHA)
depth 1 Clone depth. Set to 0 or full for full history

10

Self-hosted runners

Every job runs on a runner you register. Runners connect via WebSocket, poll for jobs, and report results back in real-time.

How runners work

  1. Generate a registration token in Repository → Settings → Actions → Runners.
  2. Start the runner binary with --token and --labels. It registers itself and gets a long-lived JWT.
  3. The runner opens a persistent WebSocket connection to OriginHub. It sends heartbeats every 30 s.
  4. When a workflow triggers a job, OriginHub picks the first idle runner whose labels match runs-on: and dispatches the job over the WebSocket.
  5. The runner executes steps in order, streams logs back live, and reports completion.

Runner statuses

Status Meaning
online Idle and ready to accept jobs
busy Currently executing a job
offline Not connected or no heartbeat in 60 s

Matching runners with runs-on:

A runner is selected when all labels in the job's runs-on: list are present on the runner. Register multiple runners with different label sets to route jobs to specific machines (e.g. linux, gpu, arm64).

Start command

{{ actionsRunnerCmd }}

--executor can be shell (default) or docker. The registration token is single-use but the runner JWT it generates is valid for 1 year.
